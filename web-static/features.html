What it surfaces

Three answers reviewers actually need.

Every Shockwave report leads with the questions you'd ask out loud in review — answered from the graph, ranked by risk.

01 · Impact

Blast radius

The transitive set of definitions that depend on your change — following calls and inheritance across files, not just the local module. Ranked by fan-in and proximity, grouped by file, with each node's hop distance from the change.

02 · Reachability

Exposure

Is the change reachable from a public entry point — a route, an API handler, a CLI command? Shockwave traces the exact call path from the surface to your change, so you know whether a break is externally observable or strictly internal. It's the same reachability question security triage asks, applied to change risk.

03 · Coverage

Untested hotspots — with the tests

High fan-in definitions in non-test code that no test calls directly: the code most likely to break silently. Shockwave doesn't just flag them — it generates a ready-to-fill pytest stub for each, so the review comment tells you exactly what to pin down before shipping.

suggested test
def test_record_once_impact():
    """Cover Blueprint.record_once.

    Flagged by Shockwave: 10 dependents, no test
    calls it directly. Changing it could break
    callers silently."""
    from flask.sansio.blueprints import record_once
    # TODO: call record_once(...) and assert behavior
    raise NotImplementedError
